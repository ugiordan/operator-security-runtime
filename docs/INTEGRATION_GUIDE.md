# Integration Guide: operator-security-runtime

This guide walks you through integrating the `operator-security-runtime` library
into an existing Go-based Kubernetes operator. The library provides independent
security packages that can be adopted together or separately:

- **`pkg/saprotection`** -- A validating webhook that prevents unauthorized pods
  from using your operator's ServiceAccount.
- **`pkg/rbacscope`** -- Dynamic, namespace-scoped RBAC that grants resource
  access only where your Custom Resources exist.
- **`pkg/impersonationguard`** -- A reconciler that strips the `impersonate`
  verb from `system:aggregate-to-edit`, closing the impersonation bypass.
- **`pkg/rbacaudit`** -- A startup audit that detects impersonation and token
  request exposure in the cluster's RBAC configuration.

---

## 1. Installation

```bash
go get github.com/opendatahub-io/operator-security-runtime
```

**Prerequisites:**

- Go-based Kubernetes operator using controller-runtime v0.19+.
- `kubebuilder` or an equivalent scaffold (for RBAC markers and webhook
  configuration generation).
- cert-manager (or another certificate solution) for webhook TLS.

## 2. Add SA Protection (`pkg/saprotection`)

The SA protection webhook intercepts pod `CREATE` and `UPDATE` requests. If a
pod references a protected ServiceAccount, the webhook verifies that the
request originates from that ServiceAccount's own identity. All other callers
-- including cluster-admins -- are denied.

### 2.1 Configure Protected Identities

```go
import "github.com/opendatahub-io/operator-security-runtime/pkg/saprotection"

protectedIdentities := []saprotection.ProtectedIdentity{
    {
        Namespace:          os.Getenv("OPERATOR_NAMESPACE"),
        ServiceAccountName: os.Getenv("OPERATOR_SA_NAME"),
    },
}
```

Each `ProtectedIdentity` maps to an expected Kubernetes identity string of the
form `system:serviceaccount:<namespace>:<name>`. The `Namespace` field is used
to construct this identity; the webhook itself applies cluster-wide (subject to
the namespace selector configured below).

### 2.2 Register Webhook in main.go

```go
if os.Getenv("ENABLE_WEBHOOKS") != "false" {
    if err := saprotection.SetupPodWebhookWithManager(mgr, protectedIdentities); err != nil {
        setupLog.Error(err, "unable to create webhook", "webhook", "Pod")
        os.Exit(1)
    }
}
```

`SetupPodWebhookWithManager` registers a `CustomValidator` for `corev1.Pod`
with the manager's webhook server. The webhook path is
`/validate--v1-pod`.

### 2.3 Deploy Webhook Configuration

Create a `ValidatingWebhookConfiguration` manifest (typically generated by
kubebuilder in `config/webhook/manifests.yaml`). The critical settings are:

- **`failurePolicy: Fail`** -- Fail-secure. If the webhook is unreachable,
  pod creation is blocked rather than silently allowed.
- **`sideEffects: None`** -- The webhook does not modify cluster state.
- **`namespaceSelector`** -- Excludes system namespaces and the operator's own
  namespace to prevent deadlock on restart.

### 2.4 Kustomize Patch for Namespace Selector

Create `config/webhook/webhook_namespace_selector_patch.yaml`:

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: validating-webhook-configuration
webhooks:
  - name: vpod-v1.kb.io
    namespaceSelector:
      matchExpressions:
        - key: kubernetes.io/metadata.name
          operator: NotIn
          values:
            - kube-system
            - kube-public
            - kube-node-lease
            # The operator's own namespace MUST be excluded to prevent
            # deadlock: if the webhook pod restarts, it must not need
            # webhook approval to start again.
            - your-operator-system
```

Reference the patch in `config/webhook/kustomization.yaml`:

```yaml
resources:
- manifests.yaml
- service.yaml

patches:
  - path: webhook_namespace_selector_patch.yaml

configurations:
- kustomizeconfig.yaml
```

### 2.5 Finding Your Values

If you are unsure of your operator's ServiceAccount name or namespace, extract
them from the deployment:

```bash
# ServiceAccount name
kubectl get deployment -n <ns> <deployment> \
  -o jsonpath='{.spec.template.spec.serviceAccountName}'

# Namespace
kubectl get deployment -n <ns> <deployment> \
  -o jsonpath='{.metadata.namespace}'
```

## 3. Add RBAC Scoping (`pkg/rbacscope`)

The RBAC scoper replaces static cluster-wide resource access with dynamic,
namespace-scoped Roles. A Role and RoleBinding are created in each namespace
where a CR exists, and cleaned up when the last CR in that namespace is deleted.

### 3.1 Initialize Scoper

In `main.go`, after creating the manager:

```go
import (
    "github.com/opendatahub-io/operator-security-runtime/pkg/rbacscope"
    rbacv1 "k8s.io/api/rbac/v1"
)

rbacScoper := &rbacscope.RBACScoper{
    Client:              mgr.GetClient(),
    Scheme:              mgr.GetScheme(),
    OperatorName:        os.Getenv("OPERATOR_NAME"),
    OperatorSAName:      os.Getenv("OPERATOR_SA_NAME"),
    OperatorSANamespace: os.Getenv("OPERATOR_NAMESPACE"),
    Rules: []rbacv1.PolicyRule{{
        APIGroups: []string{""},
        Resources: []string{"secrets"},
        Verbs:     []string{"get", "list", "watch"},
    }},
}
```

Pass the scoper to your reconciler:

```go
if err = (&controller.YourReconciler{
    Client:     mgr.GetClient(),
    Scheme:     mgr.GetScheme(),
    RBACScoper: rbacScoper,
}).SetupWithManager(mgr); err != nil {
    setupLog.Error(err, "unable to create controller")
    os.Exit(1)
}
```

### 3.2 Update Reconciler

Add the `RBACScoper` field to your reconciler struct and call
`EnsureAccess` before any logic that accesses scoped resources:

```go
import "github.com/opendatahub-io/operator-security-runtime/pkg/rbacscope"

type YourReconciler struct {
    client.Client
    Scheme     *runtime.Scheme
    RBACScoper *rbacscope.RBACScoper
}

func (r *YourReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    cr := &yourv1.YourResource{}
    if err := r.Get(ctx, req.NamespacedName, cr); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    // Handle deletion (see 3.3 below)
    if !cr.DeletionTimestamp.IsZero() {
        return r.handleDeletion(ctx, cr)
    }

    // Add finalizer if not present
    if !controllerutil.ContainsFinalizer(cr, finalizerName) {
        controllerutil.AddFinalizer(cr, finalizerName)
        if err := r.Update(ctx, cr); err != nil {
            return ctrl.Result{}, err
        }
    }

    // Ensure scoped access BEFORE accessing scoped resources
    if err := r.RBACScoper.EnsureAccess(ctx, cr); err != nil {
        return ctrl.Result{}, fmt.Errorf("ensuring scoped access: %w", err)
    }

    // Now safe to access scoped resources in cr.Namespace
    // ...

    return ctrl.Result{}, nil
}
```

### 3.3 Add Finalizer for Cleanup

The finalizer ensures that scoped RBAC resources are cleaned up when the CR is
deleted. The scoper uses OwnerReferences, so if multiple CRs share a namespace,
deleting one CR only removes its OwnerReference -- the Role and RoleBinding
persist until the last CR is removed.

```go
const finalizerName = "your-operator.example.com/scoped-access-cleanup"

func (r *YourReconciler) handleDeletion(
    ctx context.Context,
    cr *yourv1.YourResource,
) (ctrl.Result, error) {
    if controllerutil.ContainsFinalizer(cr, finalizerName) {
        if err := r.RBACScoper.CleanupAccess(ctx, cr); err != nil {
            return ctrl.Result{}, fmt.Errorf("cleaning up scoped access: %w", err)
        }
        controllerutil.RemoveFinalizer(cr, finalizerName)
        if err := r.Update(ctx, cr); err != nil {
            return ctrl.Result{}, err
        }
    }
    return ctrl.Result{}, nil
}
```

### 3.4 Watch Owned RBAC Resources

Register the controller to watch Role and RoleBinding resources owned by your
CR. This ensures that if someone deletes or modifies the scoped RBAC resources
externally, the controller re-reconciles and restores them:

```go
func (r *YourReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&yourv1.YourResource{}).
        Owns(&rbacv1.Role{}).
        Owns(&rbacv1.RoleBinding{}).
        Complete(r)
}
```

### 3.5 Add RBAC Markers

The operator needs permission to manage Roles and RoleBindings. Add these
kubebuilder markers to your reconciler file:

```go
// +kubebuilder:rbac:groups=rbac.authorization.k8s.io,resources=roles,verbs=get;list;watch;create;update;delete;escalate
// +kubebuilder:rbac:groups=rbac.authorization.k8s.io,resources=rolebindings,verbs=get;list;watch;create;update;delete
```

The `escalate` verb on Roles is required because the operator creates Roles
that grant resource access -- without `escalate`, Kubernetes RBAC escalation
prevention would block the Role creation.

### 3.6 Remove Scoped Resources from ClusterRole

Remove the entries for resources managed by the scoper from your operator's
static ClusterRole manifest. After integration, resource access is granted
dynamically per-namespace by the scoper. Keeping the static entry would defeat
the purpose of scoping.

```yaml
# BEFORE (remove scoped resource rules)
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch"]

# AFTER: no scoped resource rules in ClusterRole
```

See Section 5 for a safe migration strategy.

## 4. Configuration

### Environment Variables

| Variable | Description | Example |
|---|---|---|
| `OPERATOR_NAMESPACE` | Namespace where the operator is deployed | `my-operator-system` |
| `OPERATOR_SA_NAME` | Name of the operator's ServiceAccount | `my-operator-controller-manager` |
| `OPERATOR_NAME` | Operator name (used for Role/RoleBinding naming) | `my-operator` |
| `ENABLE_WEBHOOKS` | Set to `"false"` to disable webhooks (e.g., in tests) | `"true"` |

Use the Kubernetes downward API to inject `OPERATOR_NAMESPACE` automatically:

```yaml
env:
  - name: OPERATOR_NAMESPACE
    valueFrom:
      fieldRef:
        fieldPath: metadata.namespace
```

### Custom Rules

The `Rules` field accepts any `[]rbacv1.PolicyRule`, controlling exactly what
the scoped Role grants. Use read-only secrets access for operators that only
consume secrets:

```go
Rules: []rbacv1.PolicyRule{{
    APIGroups: []string{""},
    Resources: []string{"secrets"},
    Verbs:     []string{"get", "list", "watch"},
}}
```

For operators that need access to multiple resource types:

```go
Rules: []rbacv1.PolicyRule{
    {
        APIGroups: []string{""},
        Resources: []string{"secrets"},
        Verbs:     []string{"get", "list", "watch"},
    },
    {
        APIGroups: []string{""},
        Resources: []string{"configmaps"},
        Verbs:     []string{"get", "list"},
    },
}
```

### Cross-Controller CRD Access

The `Rules` field is not limited to core resources. If your controller needs
scoped access to Custom Resources managed by a different controller, include
the CRD's API group in the rules:

```go
Rules: []rbacv1.PolicyRule{
    {
        APIGroups: []string{""},
        Resources: []string{"secrets"},
        Verbs:     []string{"get", "list", "watch"},
    },
    {
        APIGroups: []string{"mlflow.example.com"},
        Resources: []string{"mlflowservers"},
        Verbs:     []string{"get", "list", "watch"},
    },
}
```

When a FeatureStore CR is created in namespace X, the scoper grants the
FeatureStore SA access to both secrets and MLFlow CRs in that namespace.
This works when both CRs co-locate in the same namespace (the common pattern
in multi-tenant platforms where users create all related CRs in their project
namespace).

**Limitation:** Access is scoped to namespaces where the triggering
controller's own CRs exist. If the target CRs live in a different namespace,
use a static ClusterRole for that cross-namespace access instead.

### Multiple ServiceAccounts

If your operator deploys multiple components with separate ServiceAccounts,
add each to the `ProtectedIdentities` slice:

```go
protectedIdentities := []saprotection.ProtectedIdentity{
    {Namespace: ns, ServiceAccountName: "my-operator-controller-manager"},
    {Namespace: ns, ServiceAccountName: "my-operator-data-plane"},
}
```

## 5. Migration from Static ClusterRole

Migrating from a cluster-wide resource ClusterRole to scoped Roles can be done
safely in three steps with no downtime.

### Step 1: Deploy New Code (Keep Existing ClusterRole)

Deploy the operator with `RBACScoper` integrated but leave the existing
resource entries in the static ClusterRole. The ClusterRole acts as a fallback
while scoped Roles are being created. Both paths grant access, so there is no
disruption.

### Step 2: Verify All CRs Have Been Reconciled

Confirm that scoped Roles exist in every namespace where a CR is present:

```bash
# List all CRs across namespaces
kubectl get <your-cr> --all-namespaces

# Verify scoped Roles exist
kubectl get roles -A | grep <your-operator>-scoped-access

# Verify scoped RoleBindings exist
kubectl get rolebindings -A | grep <your-operator>-scoped-access-binding
```

Each namespace with a CR should have a corresponding Role and RoleBinding.

### Step 3: Remove Scoped Resources from Static ClusterRole

Once all namespaces are covered by scoped Roles, remove the scoped resource
rules from the static ClusterRole manifest and redeploy. From this point, the
operator only has resource access in namespaces where a CR exists.

## 6. Complementary RBAC Hardening

The library addresses pod-level SA hijacking and secrets scoping. Two additional
attack vectors should be addressed via static RBAC policy.

### TokenRequest API

An attacker with `create` access to `serviceaccounts/token` in the operator's
namespace can mint tokens for the operator SA without creating a pod. Restrict
this with a ClusterRole that limits token creation:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: restrict-operator-token-creation
rules:
  - apiGroups: [""]
    resources: ["serviceaccounts/token"]
    verbs: ["create"]
    # Bind this only to the operator's own SA or specific controllers
    # that legitimately need to mint tokens.
```

Ensure that no broad ClusterRoleBindings grant `create` on
`serviceaccounts/token` to untrusted subjects in the operator's namespace.

### Impersonation

The default Kubernetes `system:aggregate-to-edit` ClusterRole grants
`impersonate` on `serviceaccounts`, which aggregates into `edit` and `admin`
roles. This means **any namespace editor can bypass the SA protection webhook**
by running `kubectl --as=system:serviceaccount:<ns>:<sa>`.

**Automated mitigation:** Use `pkg/impersonationguard` to strip the
`impersonate` verb from `system:aggregate-to-edit`. Register the reconciler in
your operator's `main.go` (see the Quick Start in README.md or Section 2 for
the pattern).

**Additional hardening:** Deploy the ValidatingAdmissionPolicy from
`config/validatingadmissionpolicy/` to prevent non-system users from creating
new RBAC resources that grant `impersonate`:

```bash
kubectl apply -f config/validatingadmissionpolicy/
```

This requires Kubernetes 1.30+ (OCP 4.17+).

## 7. Granting Workload ServiceAccounts Access to Resources

> **Note:** This pattern is distinct from the library's core mission of
> *restricting* operator permissions. It is documented here because operators
> that manage workloads often need to grant those workloads access to cluster
> resources (e.g., a notebook SA accessing MLflow endpoints).

When your operator creates workloads (Pods, Jobs, StatefulSets) that run under
their own ServiceAccount, those workloads may need access to resources in the
same namespace. Rather than granting cluster-wide permissions, create
namespace-scoped RoleBindings tied to the workload CR lifecycle — the same
pattern used by `RBACScoper`, but for the workload SA instead of the operator SA.

```go
func (r *MyReconciler) ensureWorkloadAccess(ctx context.Context, cr *myv1.MyCR) error {
    roleName := "my-operator-workload-access"
    workloadSA := cr.Spec.ServiceAccountName // SA the workload runs under
    targetNS := cr.Namespace

    // Create or update the Role with the permissions the workload needs
    role := &rbacv1.Role{
        ObjectMeta: metav1.ObjectMeta{
            Name:      roleName,
            Namespace: targetNS,
        },
    }
    _, err := controllerutil.CreateOrUpdate(ctx, r.Client, role, func() error {
        role.Rules = []rbacv1.PolicyRule{{
            APIGroups: []string{""},
            Resources: []string{"secrets"},
            Verbs:     []string{"get", "list"},
        }}
        return controllerutil.SetOwnerReference(cr, role, r.Scheme)
    })
    if err != nil {
        return fmt.Errorf("ensuring workload Role: %w", err)
    }

    // Create or update the RoleBinding
    rb := &rbacv1.RoleBinding{
        ObjectMeta: metav1.ObjectMeta{
            Name:      roleName + "-binding",
            Namespace: targetNS,
        },
    }
    _, err = controllerutil.CreateOrUpdate(ctx, r.Client, rb, func() error {
        rb.RoleRef = rbacv1.RoleRef{
            APIGroup: "rbac.authorization.k8s.io",
            Kind:     "Role",
            Name:     roleName,
        }
        rb.Subjects = []rbacv1.Subject{{
            Kind:      "ServiceAccount",
            Name:      workloadSA,
            Namespace: targetNS,
        }}
        return controllerutil.SetOwnerReference(cr, rb, r.Scheme)
    })
    if err != nil {
        return fmt.Errorf("ensuring workload RoleBinding: %w", err)
    }

    return nil
}
```

**Key differences from `RBACScoper`:**

| | Operator scoping (`RBACScoper`) | Workload access (this pattern) |
|---|---|---|
| **Purpose** | Restrict operator SA permissions | Grant workload SA permissions |
| **Subject** | Operator's ServiceAccount | Workload's ServiceAccount |
| **Lifecycle** | Tied to CR via OwnerReference | Tied to CR via OwnerReference |
| **Cleanup** | Automatic (OwnerReference GC) | Automatic (OwnerReference GC) |

Because `controllerutil.SetOwnerReference` ties the Role and RoleBinding to the
CR, Kubernetes garbage collection handles cleanup automatically when the CR is
deleted — no finalizer needed.

**When to use this pattern:**

- Your operator creates workloads that need access to namespace-scoped resources
- You want to avoid granting cluster-wide permissions to workload SAs
- The workload SA and the target resources are in the same namespace

**When NOT to use this pattern:**

- Cross-namespace access (use `ClusterRoleBinding` or consider whether the
  architecture should be redesigned)
- The workload already has the permissions it needs via existing roles

## 8. Troubleshooting

### RBAC Propagation Delay

After `EnsureAccess` creates a Role and RoleBinding, there may be a brief
delay (typically under one second) before the API server authorizes requests
using the new Role. If your reconciler accesses scoped resources immediately
after ensuring access and encounters a `403 Forbidden`, requeue with a short
delay:

```go
secretList := &corev1.SecretList{}
if err := r.List(ctx, secretList, client.InNamespace(cr.Namespace)); err != nil {
    log.Error(err, "secrets RBAC may not have propagated yet, requeuing")
    return ctrl.Result{RequeueAfter: 2 * time.Second}, nil
}
```

### Certificate Issues

The validating webhook requires TLS certificates. Use cert-manager with a
`Certificate` resource that targets the webhook Service. Ensure the
`caBundle` field in the `ValidatingWebhookConfiguration` is injected
automatically (cert-manager's `cert-manager.io/inject-ca-from` annotation).

### Webhook Deadlock

If the operator namespace is not excluded from the webhook's
`namespaceSelector`, the webhook blocks its own pod from starting after a
restart. The operator pod references the protected SA, the webhook denies the
request because the webhook server is not running to approve it, and the pod
never starts.

Always exclude the operator namespace from the `namespaceSelector` as shown in
Section 2.4. The operator namespace is protected by other means (RBAC on the
namespace itself, network policies, etc.).

### RoleBinding Drift

The `RoleRef` field on a `RoleBinding` is immutable after creation. If someone
externally modifies the `RoleRef` (which Kubernetes rejects) or deletes and
recreates the `RoleBinding` with a different `RoleRef`, the scoper detects the
`Invalid` error, deletes the stale `RoleBinding`, and recreates it with the
correct `RoleRef`. No manual intervention is needed.

### Webhook Bypassed by Mutating Webhooks

Kubernetes runs mutating admission webhooks before validating webhooks. If a
mutating webhook changes the `serviceAccountName` field after this validating
webhook approves the pod, the change would not be caught unless webhook
reinvocation is enabled. Audit your cluster for mutating webhooks that modify
pod ServiceAccount fields.
